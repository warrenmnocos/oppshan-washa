Warren sometimes runs **two Claude sessions on this repo at once** (this session: one on the AWS deployment + UI, a parallel one on the GraalVM PGO pipeline). They share a single working tree and git index.

**Rule:**
- **Commit only explicit paths: `git commit -- <path> [<path>...]`.** A bare `git commit` commits the WHOLE index, which can include files the OTHER session has staged. Lesson: a bare `git commit` swept the other session's staged `config/Person.java` into a docs commit; it had to be undone (`git reset --soft HEAD~1`) and re-committed with explicit paths. A narrow "no secrets staged" grep is not enough — it won't catch an unrelated staged source file. Always check `git diff --cached --name-only` is exactly your files, or just commit explicit paths.
- **Don't edit files the other session owns — and the domain varies by period.** When told another session handles X, stay off those surfaces. *PGO period:* `scripts/graalvm-pgo/**`, the `%pgo.*` blocks in `application.properties`, the `native-release-pgo*` Maven profiles, `cd.yml`'s build path, `docs/native-image.md`, `docs/diagrams/**`. *Backend-test/infra period (2026-06-30):* `src/test/**` test fixes, the `%test.*` blocks in `application.properties` (e.g. they added `devservices.reuse=false`), and the Tokyo→Singapore relocation + AWS env-var namespacing to `OPPSHAN_WASHA_*` across `infra/`, `README.md`, `CLAUDE.md`. When restyling a shared file (e.g. reorganizing `application.properties` into sections), preserve their blocks verbatim and leave their sub-headers in place.
- **Don't run a full test suite concurrently with the other session.** Two `./mvnw test` runs share the Dev Services Postgres/Keycloak containers and the CPU; running them at once causes SocketTimeouts plus cross-session row collisions (duplicate `year_month` keys, stale `fx_rate` pollution) that *look* like real failures but are contention/stale-data, not your change. If you must verify while they're active, isolate with a fresh container (`TESTCONTAINERS_REUSE_ENABLE=false ./mvnw test`) and treat the result as provisional, or wait for their work to land and run one clean pass.
- **Commit your scoped, independently-verified work even when the shared suite is red because of the other session's in-flight fixes.** Don't block your task on their domain. Verify what you *can* in isolation (your files compile, augmentation/packaging/frontend-build pass), commit explicit paths, and let their fixes + CI green the full suite. Warren may say so outright ("commit, another session will fix the tests"). Re-run the full clean verification once their work lands — this period: the deps bump was committed against a red suite, then went 107/107 + native `BUILD SUCCESS` after their test fixes merged.
- **Shared docs both sessions touch** (`CLAUDE.md`, `README.md`, `.gitignore`): keep edits minimal and additive; re-Read before editing in case the other session changed them between your turns.

**Why:** avoids clobbering or mis-attributing the parallel session's work, and keeps each commit cleanly scoped to one session's task.

**How to apply:** treat the working tree as shared — never assume the index holds only your changes; scope every commit by path.
